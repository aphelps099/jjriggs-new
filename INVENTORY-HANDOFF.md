## 2. The workbook (Andrew's interface)

Five tabs: **READ ME FIRST**, **Tractors** (27 rows), **Mowers** (15), **Implements** (82, variant schema), **Accessories** (8, seeded verbatim from Andrew's July 8 email — ballast, block heater, cutting edge, 3rd-function hydraulics, rear remotes, bucket hooks, backhoes, backhoe thumbs).

Key schema ideas:
- `status` dropdown (Active/Hidden/Discontinued) — nobody ever deletes a row.
- Implements use **style_name + size_label**: rows sharing a style_name render as ONE product card with a sizes table. Andrew's "5 grapple styles × 2 sizes" = 10 rows. Width units are explicit (`4'` vs `60"`) because site data mixes feet (`width`) and inches (`widthIn`).
- `image_current` is read-only truth; `new_photo_url` is where Andrew pastes Drive links.
- Fields the sheet doesn't carry (`c1/c2/c3`, `hpRaw`, `page`, `slug`, `id`) are preserved from existing items by the builder on import — do not add them to the sheet.

Regenerate the CSVs any time with `node tools/export-to-sheet.mjs [outDir]`. The **round-trip identity test** is the pipeline's health check: export → import into /admin/ → diff must be zero (only known exception: 9 one-time name normalizations moving trailing sizes to leading, e.g. `Large Front Grapple (70")` → `70" Large Front Grapple`).

## 3. The /admin/ builder

Five steps: **Import** (drop tab CSVs or fetch published-sheet CSV links) → **Validate & Diff** (enum/dupe checks + field-level diff vs live data; dirty data cannot publish) → **Missing info** (per-row AI extraction) → **Photos** (≤5 per product, browser-side resize to 1400px/JPEG; first = primary image, extras → tractor gallery; photo-only publishes allowed) → **Publish** (commits to a `builder/*` branch via GitHub API; Action regenerates pages; Cloudflare builds branch preview; human merges).

Keys (Settings ⚙, localStorage only, never committed):
- GitHub fine-grained PAT — repo `aphelps099/jjriggs-new` only, Contents R/W (current one expires 2026-08-07).
- Anthropic API key — powers "Fetch specs"; model dropdown defaults `claude-sonnet-5` (~3¢/product).

Verification status: round-trip import ✅ · photo flow ✅ (browser test) · proxy allowlist ✅ · AI extraction ✅ (live key test; returned honest `null` for unstated hpMin) · **GitHub publish ⏳ — code follows the documented API but couldn't be pre-flighted from the dev sandbox (its proxy blocks arbitrary GitHub writes); the first real Publish from Aaron's browser is the live test.** If it errors, the publish log shows verbatim GitHub responses.

Known limits (v1, intentional): sheet-blank cells can't *clear* preserved extra fields; Hidden/Discontinued rows simply drop from generated files (history lives in the sheet + git); the Action regenerates implement pages + sitemap only (extend `regenerate.yml` when tractor/mower pages become generated).

## 5. Facebook / "On the Lot" (scoped, NOT built)

The original ask: pull Andrew's Facebook page + Marketplace listings onto the site as "On the Lot" cards. Findings from the July 8 scoping (verified then):

- **Marketplace has NO public API** (partner-only). Scraping violates Meta ToS and risks the account. Hard no.
- **Page posts ARE fully accessible**: a Meta app in dev mode (Andrew as app admin — no App Review needed for your own page) + a never-expiring Page token can read `/JJRIGGSEQUIPMENT/feed`.
- Decided architecture: **Facebook is an input feed, never the source of truth.** Scheduled sync (GitHub Action or Worker cron) reads hashtagged posts (`#onthelot`), downloads photos (FB CDN URLs expire — never hotlink), and writes rows into an "On the Lot" sheet tab → normal pipeline publishes.
- **Staleness (the "what if it sells on FB" problem) — four layers**: (1) cards keyed on post ID: post deleted or edited to say SOLD → card drops next sync; (2) TTL ~30 days auto-hide + flag; (3) every card shows "Listed <date> — call to confirm: 509-738-2985"; (4) sync failure alerts loudly (last-success timestamp) instead of serving stale.
- Conservative parsing: price only from an explicit `$`, title from first line, rest verbatim post text. Never invent.
- **Reverse direction is the richer play now that the builder exists**: a product entry → studio motion card (the `/studio/` Inventory Builder already generates full unit videos from typed specs) → post TO the page via `pages_manage_posts` (the easy API direction). One Meta app serves both directions.
- Prereq either way: Andrew present for ~1hr Meta app setup + token grant, and his blanket sign-off that his own FB posts may auto-publish to the site section (repo rule: nothing ships without his review).
